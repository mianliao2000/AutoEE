# 面向电力电子的 AI-native 设计编辑器 / 工程平台 —— 商业计划书（草稿）

**对投资人：** AI-native power design platform / experience system  
**对用户：** Cursor for Power Electronics

---

## 背景（为什么要做这件事）

电力电子 PCB 设计是整个电子设计自动化领域里最依赖专家经验、也最难标准化的环节之一。

- 从规格定义到可制造的第一版电源板，通常要消耗 3-12 周资深工程师时间
- 设计过程天然跨域：拓扑选择、器件筛选、磁件设计、损耗分析、环路稳定性、热、EMC、布局布线、安规、制造
- 资深工程师和大公司团队之所以长期占优，核心原因在于其掌握了长期沉淀的经验、方法论和验证习惯
- 这些经验大量是可迁移、可复用的，但当前缺少一个能够系统汇总、组织、复用这些经验的平台
- 传统 EDA 工具主要承担“绘图 + 检查”平台角色，对设计决策的直接参与仍然有限
- 现有 AI EDA 公司已经证明了 AI 可以进入 PCB 工作流，但它们主要覆盖通用 PCB 的协作、代码化设计、布局布线和制造流转；电力电子真正稀缺的部分仍然是**领域物理 + 规则 + 验证 + 可落地工作流**

**我们的判断：**  
“AI + PCB”本身已不足以构成差异化叙事。更具壁垒潜力的方向，是把成熟的电力电子设计 know-how 编译成**AI-native 编辑器 + 持续在线的工程经验系统**。对于中小团队而言，这意味着可以用软件的方式获得接近顶级工程组织的设计经验、方法论和决策质量。

**这份文档的目的：**  
给技术背景很强的 Seed 投资人一份更可信的计划书，讲清楚：

- 我们最先卖什么
- 为什么这个起步 wedge 可落地
- 长期壁垒的主要来源是什么

## 1. 产品定位与核心主张

**产品定义：**  
本项目定位为**面向电力电子的 AI-native 自研设计编辑器 / 工程平台**。系统以自研 all-in-one app 为载体，从规格、reference design 或现有设计资产出发，生成可审查、可验证、可制造的第一版设计草案，并将设计决策、验证、协作与反馈连接为闭环。

**阶段性产品原则：**  
系统采用**可解耦、可中途接管、可对话调优**的工作方式，并在同一个自研 app 中承载 end-to-end 设计流程中的关键步骤，同时保留用户在任意阶段手动修改的能力。

**产品定位：**

- 对投资人，我们是 **AI-native power design platform / experience system**
- 对用户，我们仍然可以讲 **Cursor for Power Electronics**

前者强调壁垒在自研编辑器、经验系统、验证和工作流；后者强调产品体验接近工程师协作助手。

**系统边界：**

- 公司从 Day 1 启动自研编辑器建设，产品边界建立在自有 all-in-one app 上
- 编辑器以 AI 设计为导向，逐步覆盖 end-to-end 设计流程中的关键步骤，而非停留在传统绘图工具层
- 系统不采用不可解释的完全自动化模式，保留明确的人工签核边界
- 产品可保留与常见 CAD、仿真和制造格式的导入导出兼容能力，但核心价值承载在自研编辑器与项目系统中
- 电力电子是首个重点场景，因为该领域约束密集、错误成本高、质量反馈强，适合形成高价值闭环
- 每个阶段均可 decouple，用户可在任意节点开始、暂停、修改、退出或继续
- 人工修改被视为一级能力，并被纳入系统工作流
- LLM 项目助手作为贯穿整个项目周期的持续在线协作层存在

**阶段性边界：**

- 长期方向包括从空白规格到设计交付、再到实体硬件交付的完整链路
- Phase 0 / Phase 1 在自研编辑器内优先覆盖最有价值、最常用的关键步骤
- Phase 0 / Phase 1 默认采用**基于成熟 reference design 的参数化变体生成**路径，而非从零自由生成
- 首阶段交付物为 **verified draft**，而非直接面向生产的一键式自动生成结果
- human-in-the-loop 的含义不限于最终 sign-off，而覆盖整个流程中的人工接管和校正

**Reference-design adaptation 作为起步路径的原因：**

- 现实中的电源工程师大量工作，本来就是从 TI / ADI / Infineon 等成熟参考设计出发
- 这条路径更容易建立用户信任
- 技术难度更可控，benchmark 更清楚
- 更容易把设计依据追溯到 datasheet、公式和规则来源

**经验平台的组成：**

- 规则库
- reference family 与模板体系
- 设计决策逻辑
- 验证与修正策略
- 项目助手维护的上下文能力
- 软件项目与设计服务项目沉淀的反馈数据

这意味着，用户购买的并不只是单一软件工具，而是一套持续在线、持续运行的工程经验系统。对于小公司和小团队而言，这套系统能够以产品化方式提供接近顶级工程团队的设计经验、方法论和决策支持。

**LLM 项目助手定位：**

- 项目助手定位为项目级协作层与经验系统入口
- 它持续维护项目目标、约束、优先级和修改历史
- 它在项目推进过程中负责解释方案、比较取舍并提示风险
- 它能够根据用户手动修改过的 schematic / layout / BOM 更新后续建议
- 它的作用覆盖整个项目周期中的 fine-tune 与协同决策，并以持续在线的方式服务于后续迭代

**目标工作流：**

```text
起点可以是:
规格 / reference design / 现有 BOM / 现有 schematic / 现有 PCB layout / 现有验证问题
        ↓
规格澄清:
系统可从这里开始
用户可在这里暂停 / 退出
用户可修改目标后继续交给系统
        ↓
参考路径选择:
先在成熟 reference design / 控制器族中定位候选，再决定是否需要更自由的生成
系统可从这里开始
用户可在这里停止，只拿候选方案
用户可改约束、改候选，再继续
        ↓
设计决策:
候选拓扑、主控、MOSFET、磁件、电容、电阻网络、保护支路
系统可从这里开始
用户可在这里暂停
用户可手动改 BOM / 拓扑偏好后继续
        ↓
系统级分析:
损耗、热、环路稳定性、安规间距、降额、EMC 风险
系统可从这里开始
用户可只拿分析结果，不继续生成
用户可根据分析结果改目标后继续
        ↓
AI-native 编辑器内设计生成:
原理图 → layout draft → 布线 / 规则修正 → 版本记录
系统可从现有 schematic / layout 开始
用户可在任意中间节点接管编辑
用户改完后可重新交给系统继续
        ↓
验证包输出:
ERC / DRC / 关键公式 / datasheet 引用 / 风险提示 / 人工签核清单
系统可从这里开始，针对现有设计直接输出验证
用户可停在这里做 review / sign-off
用户可修改设计后重跑验证
        ↓
工程师审查 / 修改 / 继续编排:
作为 project loop 的组成部分持续发生
        ↓
导出 / 同步:
项目包 + schematic / layout / BOM / 制造资料 + 云端版本记录
        ↓
反馈:
人工修改、打样、实验室测试、返修与失效结果回流
```

**核心判断：**  
本项目的核心价值不在于“AI 自动出板”这一单点能力，而在于将**规格、reference、拓扑、器件、规则、验证、人工 sign-off 与制造反馈**组织为一个可持续运行的工程经验系统，并通过自研编辑器把这些能力统一交付给用户。闭环一旦成立，平台将从提效工具演进为工程基础设施。

### 可解耦工作流（Decoupled Workflow）

可解耦工作流构成系统的核心设计原则之一，而非附带的体验优化。

**系统支持的起点：**

- 从规格开始
- 从 reference design adaptation 开始
- 从已有 BOM 开始
- 从已有 schematic 开始
- 从已有 PCB layout 开始
- 从已有验证问题 / 返修问题开始

**系统支持的中止点：**

- 生成候选拓扑后停止
- 生成 BOM 后停止
- 生成 schematic draft 后停止
- 生成 layout draft 后停止
- 输出验证包后停止

**系统支持的恢复方式：**

- 用户手动改完再继续
- 用户与 LLM 项目助手讨论方案后继续
- 用户只使用某一段能力，不使用全流程

**对用户的使用边界：**

- 用户可以仅使用 spec clarification 能力
- 也可以仅使用 BOM intelligence
- 也可以仅使用 schematic / layout / verification 中的单段能力
- 用户在自研编辑器内对 PCB layout、schematic、BOM 的手动修改不会使项目脱离系统；相关修改会继续被纳入项目上下文

**长期路线（保留大叙事）：**

1. **从电源板家族切入**  
   从同步 Buck 等最可重复、最可验证、最容易形成数据闭环的电源家族开始。

2. **升级为电力电子设计决策系统**  
   把器件选择、磁件设计、补偿网络、热、EMC、安规、布局布线和验证联成可迭代系统，并服务于电车、数据中心、芯片、储能、通信等高价值行业。

3. **扩展到机电协同与团队协作平台**  
   在电子设计基础上，逐步加入机械 CAD、云存储、自动同步和团队协作能力，形成更通用的 engineering design platform。

## 2. 分模块可行性分析（诚实版）

每个模块按三档标注：🟢 现阶段可工程化 / 🟡 有难度但可逐步攻克 / 🔴 中长期研究方向  
**Seed 轮承诺范围：只承诺绿 + 一部分黄，不承诺红。**

### 2.1 规格澄清与需求结构化 🟢
- 大语言模型非常适合和工程师对话，澄清功率、电压等级、效率、成本、尺寸和保护要求
- **技术方案**：大语言模型 + 电力电子知识库检索，把自然语言转成结构化设计约束
- **Phase 0 目标**：把规格变成后续工具和规则系统可消费的输入
- **解耦意义**：用户可以只使用这一层，而不进入后续生成流程

### 2.2 Reference design 匹配与拓扑选择 🟡
- 输入输出范围、隔离需求、功率等级、效率目标都可以映射到候选参考设计与拓扑家族
- **技术方案**：规则树 + reference family 检索 + 有约束的模型排序
- **Phase 0 目标**：优先推荐成熟参考路径，降低从零生成带来的不确定性
- **解耦意义**：用户可以只拿方案比较结果，自己继续手动设计

### 2.3 BOM / 元器件选择 🟢
- DigiKey、Mouser、LCSC 等都提供器件信息与库存价格数据
- **技术方案**：
  - 参数化检索：耐压、导通电阻、封装、温度范围、库存、价格
  - 结合 datasheet 解析：补齐开关损耗、热阻、温度系数等关键参数
  - 做 alternates、availability、lifecycle 和成本综合排序
- **Phase 0 目标**：BOM intelligence 成为首个明确用户价值点
- **解耦意义**：用户可以只调用 BOM intelligence，不必生成 schematic / layout

### 2.4 损耗与热估算 🟢
- 导通损耗、开关损耗、磁芯损耗、铜损都有成熟公式
- **技术方案**：数值计算库 + 器件参数数据库 + 模块化拓扑模型
- **Phase 0 目标**：先覆盖常见电源家族的大部分一阶分析
- **解耦意义**：用户可拿已有 BOM 或原理图来直接做分析

### 2.5 环路设计与稳定性分析 🟡
- 常见拓扑有成熟的小信号模型和补偿思路
- **技术方案**：模板化传递函数 + 数值求解 + 自动给出建议补偿网络
- **Phase 0 目标**：输出建议与风险提示，不承诺复杂拓扑全自动闭环
- **解耦意义**：用户可在分析层开始，不必先重走前面的设计流程

### 2.6 安规、降额与规则生成 🟢
- 电压等级、隔离边界、应用场景和标准要求都可以映射为规则
- **技术方案**：规则表 + 条件映射，自动生成编辑器内规则、验证约束与 sign-off 检查清单
- **Phase 0 目标**：这是与通用 PCB AI 拉开差距的核心能力之一
- **解耦意义**：用户可对已有设计单独跑规则和 sign-off 检查

### 2.7 原理图与 layout draft 生成 🟡
- 选定 reference path 后，连接关系和外围网络有很强模板性
- **技术方案**：自研编辑器中的 reference design 模板库 + 参数填充 + 支路选择 + 布局草案生成
- **Phase 0 目标**：生成 engineer-reviewable 的原理图和 layout draft，作为后续审查和迭代基础
- **解耦意义**：用户可以从已有 schematic / layout 开始，也可以自己手动修改后再继续

### 2.8 Router 接入与后处理 🟢
- 布线与后处理能力可以在自研编辑器中逐步内置，并按阶段接入外部求解能力
- **技术方案**：自研 editor 内的约束驱动布线编排 + 电力电子专用后处理
- **Phase 0 目标**：先把设计意图、规则约束和人工修改闭环打通
- **解耦意义**：用户可以只调用布线或后处理，不必全流程自动化

### 2.9 EMC / 电源完整性 / 高频寄生效应 🔴
- 这是最难形成高置信度自动化的部分之一
- **技术方案**：
  - 早期：规则引擎 + 风险识别
  - 中期：开源求解器 + SPICE / 场求解协同
  - 长期：代理模型替代部分昂贵仿真
- **说明**：这部分属于长期路线，不进入 Seed MVP 承诺

### 2.10 多物理场系统级闭环 🔴
- 涉及热、EM、PI、制造偏差与实验室反馈的联合优化
- **说明**：这是长期最有价值的方向之一，暂不纳入 Phase 0/1 的承诺范围

## 3. 技术架构建议

**架构原则：公司从 Day 1 建设自研 AI-native 编辑器 / all-in-one app，并以此作为产品边界。**  
**补充原则：首版覆盖最有价值的关键步骤，原生支持 decoupled workflow、manual override、云端同步与项目级对话上下文。**

```text
┌──────────────────────────────────────────────┐
│ AI-native 编辑与交互层                          │
│ 自研 schematic / layout editor、review UI、   │
│ 人工修改、AI 生成、项目助手交互                 │
└──────────────────────────────────────────────┘
                      ↓
┌──────────────────────────────────────────────┐
│ 项目会话与上下文层                             │
│ LLM 项目助手、项目记忆、约束历史、修改历史、     │
│ 团队评论与决策记录                              │
└──────────────────────────────────────────────┘
                      ↓
┌──────────────────────────────────────────────┐
│ 设计编排与验证层                               │
│ 规格澄清、reference 匹配、设计决策、验证编排、 │
│ 任意阶段开始 / 停止 / 重跑 / 跳转               │
└──────────────────────────────────────────────┘
                      ↓
┌──────────────────────────────────────────────┐
│ 电子设计引擎与经验系统层                       │
│ 拓扑/器件选择、损耗分析、补偿建议、规则生成、   │
│ reference family、设计逻辑、验证与修正策略     │
└──────────────────────────────────────────────┘
                      ↓
┌──────────────────────────────────────────────┐
│ 云存储 / 协作 / 数据飞轮层                     │
│ 项目版本、自动同步、权限控制、训练数据、评测集、 │
│ 实验室反馈、benchmark harness                  │
└──────────────────────────────────────────────┘
```

**自研编辑器作为产品边界的原因：**

- AI 设计与人工修改必须共享统一的数据模型和交互语义
- end-to-end 关键步骤需要在同一个 app 内被编排、审查和恢复
- 云端同步、团队协作与后续机械 CAD 扩展，都要求产品从一开始就建立在自有平台边界上
- 软件业务与设计服务业务也需要共享同一套工作台与项目系统

**经验系统的结构化承载：**

- 规则库
- reference family 与模板库
- 设计决策逻辑
- 验证与修正策略
- 项目助手维护的上下文能力
- 内部设计服务项目与外部用户项目沉淀的数据反馈

**系统技术要求：**

- 每个阶段都必须有独立 I/O
- 每个阶段输出都必须能被人工修改后重新导入
- 系统必须保留阶段间上下文、修改历史和可追溯性
- 编辑器内的人工修改必须自动写回项目上下文与后续流程
- 云端版本记录、权限控制和自动同步需要成为系统底层能力
- 训练数据与评测数据的采集需要建立在 opt-in、匿名化和聚合化前提上
- 机械 CAD 与机电协同应作为后续可扩展方向纳入架构预留

**结论：**  
产品边界是自研编辑器 / all-in-one app 及其承载的 `spec ↔ draft ↔ edit ↔ verify ↔ collaborate ↔ continue` 项目回路，而非第三方 CAD 的上层插件。

## 4. 数据策略（护城河）

**核心判断：**  
本项目更具价值的数据资产，来自**训练数据回路 + 电力电子设计评测回路**的组合，而不是单纯扩大企业私有设计样本数量。

**优先积累的数据类型如下：**

| 类型 | 示例 | 价值 |
|------|------|------|
| **设计意图** | 规格、目标函数、优先级、应用场景 | 训练需求理解与设计决策 |
| **设计产物** | 原理图、PCB、BOM、规则、版本演化 | 训练结构化生成与布局模式 |
| **验证结果** | 损耗、热、环路、ERC/DRC、风险提示 | 训练评分、排序和修正能力 |
| **人工修正** | 工程师改了什么、为什么改 | 训练从 first-pass 到可交付的改进能力 |
| **现实反馈** | 打样、点亮、返修、热像、EMI、失效案例 | 构成最强的闭环壁垒 |
| **交互与协作数据** | 对话轨迹、评论、签核记录、团队版本演化 | 训练项目助手与协作编排能力 |

**更现实的数据来源分层：**

1. **公共 / 开源设计与参考设计**  
   用于理解语法、常见结构、器件组合和拓扑模板。

2. **内部设计服务与 design partner 项目**  
   这是公司最早、最稳定的一手高质量经验来源与训练样本来源。

3. **显式 opt-in 的用户与企业项目数据**  
   只有在客户明确同意的情况下，才进入训练或评测回路。

4. **匿名化、聚合化的交互轨迹与验证结果**  
   优先保留“如何修改”“为什么判错”和“如何协作”的抽象信号，避免依赖原始私有设计全文。

5. **自建仿真、benchmark harness 与测试反馈**  
   这是早期最现实、也最可控的 moat 形成方式。

**数据飞轮如何形成：**

- 用户和项目越多，平台积累的需求、修改轨迹、验证结果、测试反馈和协作记录就越多
- 在 opt-in、匿名化和聚合化前提下，这些数据会持续进入训练与评测回路
- 更强的模型、规则引擎、排序器和项目助手会带来更高的 first-pass quality 和更低的人工修改量
- 更好的结果会带来更多软件用户、设计服务项目和企业部署，形成持续增强的正反馈

**不将“所有客户数据进入训练集”作为主叙事的原因：**

- 企业客户首先在意的是隔离、权限和隐私
- 认真付费的团队更像 JITX enterprise / Flux enterprise 的买家，首先关心 proprietary design 的控制权
- 因此，护城河不宜建立在“默认收走全部私有数据”的叙事之上，而应建立在“高质量、经授权的数据飞轮 + power-design eval loop”之上

**补充说明：**

- 公司自身的设计服务项目，会形成质量更高、标签更完整的一手训练与评测数据
- 用户在任意阶段的手动接管和后续恢复，本身就是极有价值的数据形态
- LLM 项目助手围绕项目目标、修改原因和方案讨论形成的对话上下文，也会成为后续优化的重要输入
- 因此，系统积累的不仅是“生成结果”，也包括“项目如何被 fine-tune 到可交付状态”的轨迹
- 随着用户规模扩大，平台会在训练数据层和评测优化层同时变强

**结论：**  
护城河的核心，不在于接触更多私有板级数据，而在于更早建立一套从经验沉淀、训练数据、规则验证到人工 fine-tune 与实测反馈的双回路系统。

## 5. 竞品分析

现有 AI EDA 公司已经证明：AI 正在进入 PCB 工作流。  
但它们覆盖的重点，大多仍集中在**通用 PCB 的协作、代码化设计、布局自动化和制造流转**；电力电子中更具价值的部分，仍然是 **power-specific physics + rules + verification + workable workflow**。

| 公司 | 当前公开定位 | 主要强项 | 主要边界 |
|------|-------------|----------|----------|
| **JITX** | code-based、AI-driven PCB design platform | requirements-as-code、可编程设计自动化、existing CAD integration、enterprise-local | 更接近通用电子设计自动化平台，其核心垂直定位不在电力电子验证闭环 |
| **Flux** | describe → design → manufacture 的 AI-first ECAD | browser-native、协作、器件研究、原理图/布局/制造流、AI 体验强 | 更强在通用设计体验和协作，不以 power verification 为第一主轴 |
| **Quilter** | autonomous placement / routing / physics validation | 自动布局布线、physics-driven implementation、快速 layout 候选 | 更偏 layout implementation，不负责完整的设计意图、BOM intelligence 和系统级验证闭环 |
| **Altium** | 在现有 PCB 平台中叠加 machine learning 与设计分析 | 大生态、工程兼容性、既有用户基础 | AI 仍然主要叠加在既有工具流程上，电力电子垂直闭环并非主要叙事 |
| **Cadence** | AI-driven PCB / system multiphysics analysis | AI + 仿真 + multiphysics + 企业级流程 | 极强但更偏大型系统平台，不以轻量 wedge 进入电源工程师日常决策流 |

**对本项目的含义：**

- “AI + PCB” 已经是被验证过的赛道
- “竞争对手尚未关注 power” 并非准确表述
- 本项目的重点位于 **AI-native 编辑器、经验平台、电力电子验证层和可解耦工作流层**

## 6. 为什么我们会赢

**核心判断：**  
本项目优先建立的是**面向电力电子的 AI-native 编辑器与经验平台**，并通过软件产品与设计服务双轮驱动，把资深工程经验系统化地交付给用户。

**早期 wedge 重点包括：**

1. **self-built editor first**  
   从 Day 1 建设 AI-native 自研编辑器，让 AI 生成、人工修改、验证和协作共享同一工作台。

2. **experience-system first**  
   先把规则库、reference family、设计逻辑、验证与修正策略结构化，形成可持续运行的经验平台。

3. **reference-design adaptation first**  
   先把成熟 reference design 参数化、验证化、工程化，作为早期更稳健的产品起点。

4. **verified draft first**  
   先交付带验证包的第一版设计草案，以建立可审查、可推进的工程结果。

5. **design-service feedback loop first**  
   设计服务业务将成为真实项目交付、经验沉淀和高质量数据回流的重要来源。

6. **power-specific rules first**  
   安规、降额、隔离、保护和电流回路这些规则，比通用 PCB AI 更难做深，也更能形成差异。

7. **data-flywheel first**  
   通过经授权、匿名化、聚合化的训练数据与评测回路，持续提升模型、规则与项目助手能力。

8. **engineer-in-the-loop and decoupled workflow first**  
   用户不需要全盘接受整条自动流程，也能从任意节点获得价值。

### Trust Stack

Trust Stack 需要作为产品架构的正式组成部分呈现，用于说明系统的可信边界与治理方式。

**Trust Stack 包括以下要素：**

- 每一步都有 deterministic artifact  
  规格文档、候选拓扑、BOM 排序、原理图草案、layout draft、验证包均需留痕

- 每个关键决策都能追溯  
  能追溯到公式、datasheet、reference design 和规则来源，而非仅依赖模型判断

- 经验来源可以结构化沉淀  
  规则、reference、修正策略、测试结论都应沉淀为长期可复用资产

- Every stage is decouplable  
  每一步都可以独立开始 / 停止 / 重跑，不要求强制执行全自动流水线

- Manual override is first-class  
  人工修改是一级能力，并被视为标准工作流的一部分

- LLM 项目助手理解上下文  
  助手能够理解项目目标、约束和人工修改意图，并据此继续 fine-tune 建议

- human sign-off 是明确边界  
  平台生成 draft，工程师签核后才进入制造

- safety-critical 场景初期不覆盖  
  汽车 / 医疗 / 航空等高安全等级应用，不作为 Seed 阶段承诺

**制度化边界：**

- 系统允许并吸收用户手动修改
- 手动编辑被视为系统工作流的一部分，而非退出系统

**护城河的构成：**

- layout automation 属于组成部分，但不足以单独构成护城河
- prompt UX 可以提升体验，但不足以单独构成壁垒
- 更核心的壁垒来自 `self-built editor + experience system + power-specific rules + verification + data flywheel + conversational fine-tuning`

## 7. 商业模式

我们采用**软件产品 + 专精电子设计服务**的双轮驱动模式，早期成功不会建立在大规模 freemium 转化上。

**早期收入主线：**

- 付费 design partners 与定制设计项目
- 小团队专业订阅
- enterprise pilots 与私有部署

| 业务线 | 目标用户 | 收入形态 | 核心能力边界 |
|------|----------|----------|--------------|
| **免费版** | 学生、评估用户、社区开发者 | $0 | spec explorer、公开模板、有限 AI 额度 |
| **专业版** | 独立工程师、小团队 | $99 / 月 / 席位 | 私有项目、BOM intelligence、verified draft、导出 |
| **团队版** | 中小企业和设计公司 | $499 / 月 / 席位 | 协作、版本控制、云同步、私有规则库、更多自动化额度 |
| **企业版** | 电源公司、汽车、工业客户 | $50k-$500k / 年 | 私有部署、权限隔离、流程集成、专属支持 |
| **设计服务** | 需要快速迭代或定制开发的客户 | 项目费 / retainer | 平台辅助设计、快速迭代、定制化交付、测试与验证支持 |

**收紧免费版边界的原因：**

- 免费版可以帮助获客，但不应该承载太多私有设计价值
- 私有项目、导出、供应链集成、自动化额度和制造接口应该收费
- 这个赛道的主要收入更可能来自有明确工程价值和隐私诉求的团队
- 设计服务会成为早期验证平台价值和建立口碑的重要收入来源

**补充说明：**

- 可解耦 workflow 会降低用户试用门槛，因为用户不需要一次性替换整个设计流程
- 全程人工接管会显著降低工程团队的心理风险
- 项目级 LLM 助手会提高产品的持续使用频率，而非只在第一次生成时被调用
- 公司会积极拥抱各类 AI 工具，以提高内部设计服务与产品研发效率
- 电车、数据中心、芯片、储能、通信等行业，将成为高价值定制需求和企业付费的重要来源

**商业判断：**  
如果同一平台既能支撑软件订阅，又能显著提升设计服务交付效率，其企业价值预计将明显高于单纯按 seat 收费的通用 EDA 订阅。

### Phase 0：最小可行产品（0-6 个月，Seed wedge）

- 损耗分析
- 规则生成
- 原理图与 layout draft
- 编辑器内版本记录与人工接管

**这一轮新增的 Phase 0 边界：**

- 用户可单独调用规格澄清
- 用户可单独调用 BOM intelligence
- 用户可单独调用 schematic draft
- 用户可单独调用 layout draft

- 自研编辑器项目包
- engineer-reviewable verified draft
- 带引用和解释的验证包
- 可回灌的中间产物
- 适合人工继续编辑的 artifact
- 初始项目上下文与决策记录

**Phase 0 不承诺：**

- 完整 blank-sheet arbitrary design generation
- 全自动 EMC / PI / 多物理场闭环
- safety-critical 场景签核
- 一次性覆盖完整 end-to-end 的所有环节

### Phase 1：电源板主要拓扑覆盖（6-18 个月）

- 扩展到 Boost、Flyback、Buck-Boost、PFC、LLC
- 建立更强的 datasheet 解析、器件排序和 reference family 库
- 上线云存储、团队协作、自动同步和版本管理能力
- 强化 verification pack、LLM 项目助手和 design partner / 设计服务工作流

### Phase 2：更广行业与更复杂项目（18-36 个月）

- 扩展到更高功率 AC-DC、BLDC / FOC、三相系统
- 覆盖电车、数据中心、芯片、储能、通信等更高价值行业
- 建立更强的规则、测试反馈和定制项目交付闭环

### Phase 3：机电协同与通用工程平台（30-48 个月）

- 加入机械 CAD，与电子设计形成机电协同工作流
- 把器件选择、磁件设计、补偿网络、layout draft、仿真结果联成自动迭代回路
- 平台开始从“生成第一版”升级为“推荐更优方案”的通用 engineering design platform

### 超长期愿景（设计到实体交付）

- 自建 fab
- PCB 生产工厂
- 数控机床加工机械
- 从设计软件、设计服务延伸到硬件实体交付
- 最终形成真正的 end-to-end engineering service stack

这部分属于远期愿景，不纳入近中期产品承诺，也不写入当前融资阶段的直接交付目标。

## 11. Seed 阶段更像样的里程碑

相较于“1000 个成功制造 PCB”，更适合作为 Seed 阶段里程碑的是质量型指标。

- 20 个成功点亮或无重大返工的板
- `spec → BOM + schematic` 的中位时间
- layout draft 的人工二次修改时长
- 中级工程师使用平台后接近资深工程师 baseline 的 first-pass quality

**这一轮新增的里程碑维度：**

- 任意阶段启动能力是否稳定
- 人工修改后继续编排是否顺畅
- LLM 项目助手是否能正确吸收上下文与修改意图

**采用这些指标的原因：**

- 它们更早反映真实工程价值
- 它们更能支撑下一轮融资叙事
- 它们比单纯“制造数量”更能体现 first-pass quality

## 12. 当前建议默认策略选择

这部分不再写成模糊未决项，而给出当前默认答案。

1. **产品形态：自研 AI-native 编辑器 / all-in-one app first**  
   从 Day 1 建设自有编辑器，首版覆盖最有价值的关键步骤，并保留手动接管能力。

2. **MVP 切入点：Buck DC-DC first**  
   最可重复、最可验证、最容易形成闭环。

3. **公司形态：软件产品 + 设计服务双轮驱动**  
   软件提供规模化交付，设计服务提供高价值收入、真实项目反馈和经验沉淀。

4. **行业重点：power electronics first，随后进入高价值行业**  
   重点覆盖电车、数据中心、芯片、储能、通信等复杂度高、付费能力强的领域。

5. **地理重点：GTM 北美，工程 / 制造联动放华南**  
   愿意为效率和风险控制付费的早期客户更可能来自北美；工程和制造联动仍可深度连接中国。

6. **长期平台方向：电子设计 → 机电协同 → 设计到交付**  
   先完成电子设计平台，再扩展到机械 CAD、云协作，最终延伸到实体硬件交付。

## 13. 风险与应对

| 风险 | 严重度 | 应对方式 |
|------|--------|----------|
| 自研编辑器范围过大 | 🔴 高 | 首版只覆盖关键步骤，把完整 end-to-end 留给后续阶段逐步扩展 |
| 模型产出不稳定，工程师不信任 | 🔴 高 | 每一步附 deterministic artifact 和验证包，保持 human sign-off |
| 经验系统难以结构化 | 🔴 高 | 通过规则库、reference family、设计服务项目和修正轨迹逐步沉淀 |
| 高质量数据不足 | 🔴 高 | 依赖 reference、benchmark harness、设计服务项目、人工修正轨迹和 opt-in 数据 |
| 用户觉得流程太复杂 | 🟡 中 | 把 decoupled workflow 做成一级能力，允许用户只用其中一段 |
| LLM 助手不能正确吸收人工修改 | 🟡 中 | 把项目上下文和修改历史作为明确的系统层，减少对隐式记忆的依赖 |
| 超长期制造愿景被误解为短期承诺 | 🟡 中 | 在文档中明确写为远期愿景，不进入当前融资和产品承诺 |

> 工程设计长期依赖经验，这也是资深工程师和大公司体系持续占优的原因。更稀缺的能力，在于把这些分散的工程经验组织成可持续运行的软件系统。  
>
> 本项目定位为面向电力电子的 AI-native 自研设计编辑器 / 工程平台。系统从规格出发，优先基于成熟 reference design 生成可审查、可验证、可制造的第一版设计草案，并附带验证包供工程师签核。  
>
> 用户购买的并不只是单一软件工具，而是一套持续在线的工程经验系统。对于小公司和小团队而言，这相当于获得接近顶级工程组织的方法论、规则体系和设计决策能力。  
>
> 该系统不要求用户执行强制全自动流水线。用户可以在自研编辑器内从任意阶段进入，在任意阶段暂停，并对 schematic、PCB layout、BOM 或规则进行手动修改，之后再交回系统继续推进。  
>
> LLM 项目助手贯穿项目周期的协作层：它维护约束、解释取舍、吸收人工修改，并支持工程师持续 fine-tune 设计结果。  
>
> 长期壁垒主要来自经验系统、经授权的数据飞轮，以及从规格、reference、规则、验证、人工 fine-tune 到实测反馈的持续改进回路。  
>
> 公司的阶段性路径，是先建立电子设计平台，再扩展到机电协同和团队云协作；更远期的愿景，则是从设计延伸到实体硬件交付。

## 15. 验证计划

仅依靠展示性质的 demo 并不足以支撑论证。更合理的验证方式，是建立一套能够与人工流程和参考设计对比的 benchmark。

**验证主轴：**

1. 早期以 **自研编辑器中的 reference-design adaptation** 作为首个承诺路径
2. 首个交付重点是 **verified draft + 经验系统**
3. 阶段性重点放在 **first-pass quality、经验复用能力与数据回流能力**

**这一轮新增的验证目标：**

- 任意阶段启动能力是否可用
- 人工修改后继续编排是否稳定
- LLM 助手是否能正确吸收项目上下文与修改意图

**具体 benchmark 维度：**

- reference design benchmark 是否跑通
- first-pass quality 是否足够高
- 人工修改量是否显著下降
- BOM intelligence 是否明显优于人工初筛
- 点亮 / 返工 / 验证包完整度是否达到可用水平
- 中途接管后恢复执行是否稳定
- conversation-driven fine-tuning 后结果是否一致更新
- 中级工程师是否能在平台辅助下接近资深工程师 baseline
- 设计服务项目产生的数据是否持续改进模型、规则和项目助手表现

**建议的验证路径：**

1. **最小可行产品演示**  
   选同步 Buck reference family，给定规格，在自研编辑器中自动生成 verified draft。

2. **Reference benchmark 套件**  
   选 20 个真实参考设计，跑完整流程，对比：
   - `spec → BOM + schematic` 时间
   - layout draft 的人工修改时长
   - BOM 成本与替代料质量
   - 点亮率与重大返工数量
   - 验证包完整度

3. **解耦工作流测试场景**  
   - 从已有 reference design 开始，只做 BOM 和验证
   - 从已有 schematic 开始，只做 layout draft 和验证包
   - 用户手动改 layout 后，再让系统补充验证与风险提示
   - 用户暂停在 BOM 阶段，数天后恢复并继续设计
   - 用户与助手对话修改目标后，系统建议是否同步更新

4. **记录人工修正和失败案例**  
   把“为什么改”“为什么失败”结构化，这比单纯收集板子文件更值钱。

5. **争取真实制造与测试反馈**  
   一旦进入打样、点亮、返修、热像、EMI 测试，就开始建立真正的闭环壁垒。

**更关键的验证问题包括：**

- 中位速度是否有数量级提升
- 第一版设计是否足够接近可交付水平
- 修改量是否显著下降
- 用户是否真的愿意在系统里反复协作和 fine-tune
- 系统是否在真实工程反馈中持续变好
- 平台是否真的缩小了小团队与大公司之间的经验差距
