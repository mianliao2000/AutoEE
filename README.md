# 面向电力电子的 AI-native 设计编辑器 / 工程平台 —— 商业计划书（升级草稿）

**对投资人：** AI-native power design platform / experience system  
**对用户：** Cursor for Power Electronics

---

## 背景（为什么要做这件事）

电力电子 PCB 设计是整个电子设计自动化领域里最依赖专家经验、也最难标准化的环节之一。

- 从规格定义到可制造的第一版电源板，通常要消耗 3-12 周资深工程师时间
- 设计过程天然跨域：拓扑选择、器件筛选、磁件设计、损耗分析、环路稳定性、热、EMC、布局布线、安规、制造
- 资深工程师和大公司团队之所以长期占优，核心原因在于其掌握了长期沉淀的经验、方法论和验证习惯
- 这些经验大量是可迁移、可复用的，但当前缺少一个能够系统汇总、组织、复用这些经验的平台
- 传统 EDA 工具主要承担“绘图 + 检查”的平台角色，对设计决策的直接参与仍然有限
- 现有 AI EDA 公司已经证明 AI 可以进入 PCB 工作流，但真正稀缺的仍然是**领域物理 + 规则 + 验证 + 可落地工作流**

**我们的判断：**  
“AI + PCB”本身已经不足以构成差异化叙事。更具壁垒潜力的方向，是把成熟的电力电子设计 know-how 编译成**AI-native 编辑器 + 持续在线的工程经验系统**。对中小团队而言，这意味着可以用软件的方式获得接近顶级工程组织的设计经验、方法论和决策质量。

**我们的组织理念：**  
公司不仅要做 AI-native 产品，也会按 AI-native 的方式运营自身 workflow。AI 工具会被深度嵌入研发、设计服务、验证、知识管理、客户交付和内部协作中，用来放大单兵能力、压缩组织摩擦、精简团队结构。这意味着公司可以用更少但更强、更高密度的团队，在同一阶段完成传统组织往往需要更大团队才能推进的工作。

**这份文档的目的：**

- 讲清楚我们最先卖什么
- 讲清楚为什么这个起步 wedge 可落地
- 讲清楚长期技术壁垒和商业壁垒来自哪里
- 在不削弱 Seed 可信度的前提下，把长期蓝图讲完整

## 1. 产品定位与核心主张

**产品定义：**  
本项目定位为**面向电力电子的 AI-native 自研设计编辑器 / 工程平台**。系统以自研 all-in-one app 为载体，从规格、reference design 或现有设计资产出发，生成可审查、可验证、可制造的第一版设计草案，并将设计决策、验证、协作与反馈连接为闭环。

**阶段性产品原则：**

- **可解耦**：用户可以从规格、reference design、BOM、schematic、layout 或验证问题任意切入
- **可接管**：用户可以在任意阶段手动修改 schematic、layout、BOM、规则或目标
- **可对话调优**：系统支持通过持续对话解释方案、比较取舍、微调方向
- **可启发式探索**：系统不只给出默认答案，还主动提出替代方案和设计路径，帮助工程师拓宽思路

**产品定位：**

- 对投资人，我们是 **AI-native power design platform / experience system**
- 对用户，我们仍然可以讲 **Cursor for Power Electronics**

前者强调壁垒在自研编辑器、经验系统、验证和工作流；后者强调产品体验接近工程师协作助手。

**系统边界：**

- 公司从 Day 1 启动自研编辑器建设，产品边界建立在自有 all-in-one app 上
- 编辑器以 AI 设计为导向，逐步覆盖 end-to-end 设计流程中的关键步骤，而非停留在传统绘图工具层
- 系统不采用不可解释的完全自动化模式，保留明确的人工作签核边界
- 产品可保留与常见 CAD、仿真和制造格式的导入导出兼容能力，但核心价值承载在自研编辑器与项目系统中
- 电力电子是首个重点场景，因为该领域约束密集、错误成本高、质量反馈强，适合形成高价值闭环
- 每个阶段均可 decouple，用户可在任意节点开始、暂停、修改、退出或继续
- 人工修改被视为一级能力，并被纳入系统工作流
- LLM 项目助手作为贯穿整个项目周期的持续在线协作层存在

**阶段性边界：**

- 长期方向包括从空白规格到设计交付，再到实体硬件交付的完整链路
- Phase 0 / Phase 1 在自研编辑器内优先覆盖最有价值、最常用的关键步骤
- Phase 0 / Phase 1 默认采用**基于成熟 reference design 的参数化变体生成**路径，而非从零自由生成
- 首阶段交付物为 **verified draft**，而非直接面向生产的一键式自动生成结果
- human-in-the-loop 的含义不限于最终 sign-off，而覆盖整个流程中的人工接管和校正

**Reference-design adaptation 作为起步路径的原因：**

- 现实中的电源工程师大量工作，本来就是从 TI / ADI / Infineon 等成熟参考设计出发
- 这条路径更容易建立用户信任
- 技术难度更可控，benchmark 更清楚
- 更容易把设计依据追溯到 datasheet、公式和规则来源

**经验平台的组成：**

- 规则库
- reference family 与模板体系
- 设计决策逻辑
- 验证与修正策略
- 项目助手维护的上下文能力
- 软件项目与设计服务项目沉淀的反馈数据

这意味着，用户购买的并不只是单一软件工具，而是一套持续在线、持续运行的工程经验系统。对小公司和小团队而言，这套系统能够以产品化方式提供接近顶级工程团队的设计经验、方法论和决策支持。

### AI-native organization / AI-first workflow

本项目的一个重要前提是：我们不仅交付 AI-native workflow，也会把它用在公司自己的组织运营里。

**这意味着：**

- AI 工具会深度嵌入研发、设计服务、验证、知识管理、客户交付和内部协作
- 每个人默认都不是单点执行者，而是借助 AI 获得更高杠杆的 owner
- 组织目标不是简单堆人，而是形成“少而强、高密度、高协同”的 high-leverage team
- 我们追求的不是用 AI 替代人的判断，而是把人的判断放在更高价值的决策、产品化和客户交付环节

因此，即使团队人数不多，公司也有机会在早期同时推进产品、服务、验证、内容沉淀和 GTM；而随着平台扩张，组织规模虽然会增长，但增长逻辑仍然是覆盖更广的平台、行业和交付网络，而不是低效率的线性扩编。

### 启发式设计（Heuristic Design）

启发式设计是本产品的核心设计理念之一。我们不把 AI 定义为“替工程师自动做完设计”的黑盒，而是把它定义为一个能够**主动提出不同设计路径、解释 trade-off、推动方案讨论**的协作式设计伙伴。

**启发式设计的含义包括：**

- AI 会主动给出多个设计路径，而不是只给一个默认答案
- AI 会解释不同拓扑、器件、磁件、补偿网络、layout 策略之间的 trade-off
- AI 的价值不仅是提效，也是帮助工程师看见原本不会优先考虑的方案
- 工程师与 AI 的讨论过程，本身会成为学习和经验沉淀的一部分
- 系统会保留“为什么选这个方案、为什么放弃另一个方案”的上下文，而不仅保留最终结果

这使产品不只是一个生成工具，也是一套可以持续提升工程师决策质量和学习速度的工程认知系统。

**LLM 项目助手定位：**

- 定位为**协作式设计伙伴 + 教学型助手 + 项目记忆层**
- 持续维护项目目标、约束、优先级和修改历史
- 在项目推进过程中负责解释方案、比较取舍、提示风险
- 能根据用户手动修改过的 schematic / layout / BOM 更新后续建议
- 能通过启发式设计对话，帮助工程师拓宽设计空间，而不是只重复既有答案
- 覆盖整个项目周期中的 fine-tune 与协同决策，并以持续在线的方式服务于后续迭代

**目标工作流：**

```text
起点可以是:
规格 / reference design / 现有 BOM / 现有 schematic / 现有 PCB layout / 现有验证问题
        ↓
规格澄清:
系统可从这里开始
用户可在这里暂停 / 退出
用户可修改目标后继续交给系统
        ↓
启发式探索与方案讨论:
AI 主动给出多个候选路径并解释 trade-off
用户可只做方案比较，不继续生成
用户可通过对话改变目标、偏好和边界条件
        ↓
参考路径选择:
先在成熟 reference design / 控制器族中定位候选，再决定是否需要更自由的生成
        ↓
设计决策:
候选拓扑、主控、MOSFET、磁件、电容、电阻网络、保护支路
        ↓
系统级分析:
损耗、热、环路稳定性、安规间距、降额、EMC 风险
        ↓
AI-native 编辑器内设计生成:
原理图 → layout draft → 布线 / 规则修正 → 版本记录
        ↓
验证包输出:
ERC / DRC / 关键公式 / datasheet 引用 / 风险提示 / 人工签核清单
        ↓
工程师审查 / 修改 / 继续编排:
作为 project loop 的组成部分持续发生
        ↓
导出 / 同步:
项目包 + schematic / layout / BOM / 制造资料 + 云端版本记录
        ↓
反馈:
人工修改、打样、实验室测试、返修与失效结果回流
```

**核心判断：**  
本项目的核心价值不在于“AI 自动出板”这一单点能力，而在于将**规格、reference、拓扑、器件、规则、验证、人工 sign-off、启发式方案讨论与制造反馈**组织为一个可持续运行的工程经验系统，并通过自研编辑器把这些能力统一交付给用户。闭环一旦成立，平台将从提效工具演进为工程基础设施。

### 可解耦工作流（Decoupled Workflow）

可解耦工作流构成系统的核心设计原则之一，而非附带的体验优化。

**系统支持的起点：**

- 从规格开始
- 从 reference design adaptation 开始
- 从已有 BOM 开始
- 从已有 schematic 开始
- 从已有 PCB layout 开始
- 从已有验证问题 / 返修问题开始

**系统支持的中止点：**

- 生成候选拓扑后停止
- 生成 BOM 后停止
- 生成 schematic draft 后停止
- 生成 layout draft 后停止
- 输出验证包后停止
- 完成启发式方案讨论后停止

**系统支持的恢复方式：**

- 用户手动改完再继续
- 用户与 LLM 项目助手讨论方案后继续
- 用户只使用某一段能力，不使用全流程

**对用户的使用边界：**

- 用户可以仅使用 spec clarification 能力
- 也可以仅使用 BOM intelligence
- 也可以仅使用 schematic / layout / verification 中的单段能力
- 用户在自研编辑器内对 PCB layout、schematic、BOM 的手动修改不会使项目脱离系统；相关修改会继续被纳入项目上下文

**长期路线（保留大叙事）：**

1. **从电源板家族切入**  
   从同步 Buck 等最可重复、最可验证、最容易形成数据闭环的电源家族开始。

2. **升级为电力电子设计决策系统**  
   把器件选择、磁件设计、补偿网络、热、EMC、安规、布局布线和验证联成可迭代系统，并服务于电车、数据中心、芯片、储能、通信等高价值行业。

3. **扩展到机电协同与团队协作平台**  
   在电子设计基础上，逐步加入机械 CAD、云存储、自动同步和团队协作能力，形成更通用的 engineering design platform。

4. **延伸到 design OS 与 design-to-delivery stack**  
   把项目上下文、方案讨论、验证、供应链、测试、交付协同连接起来，形成更完整的工程基础设施。

## 2. 分模块可行性分析

每个模块按三档标注：🟢 现阶段可工程化 / 🟡 有难度但可逐步攻克 / 🔴 中长期研究方向  
**Seed 轮承诺范围：只承诺绿 + 一部分黄，不承诺红。**

### 2.1 规格澄清与需求结构化 🟢

- **当前可工程化方式：** 大语言模型 + 电力电子知识库检索，把自然语言需求转成结构化设计约束、假设项、待确认问题和风险边界。
- **Phase 0 能做什么：** 生成 constraint matrix、遗漏项清单、规格澄清问题和可供后续工具消费的结构化输入。
- **中期增强：** 建立行业化规格模板、应用场景库、反例提示库，并把历史项目中的常见遗漏自动带入当前对话。
- **长期技术路线：** 构建 design-intent graph，把规格、方案、验证、测试结果连接起来，形成跨项目的约束传播和需求推理能力。
- **解耦价值：** 用户可以只使用这一层，而不进入后续生成流程。

### 2.2 Reference design 匹配与拓扑选择 🟡

- **当前可工程化方式：** 规则树 + reference family 检索 + 有约束的模型排序，把功率、电压、隔离、成本、效率目标映射到候选路径。
- **Phase 0 能做什么：** 优先推荐成熟参考路径，给出候选 reference family、解释为何排除其他拓扑，并输出 engineer-reviewable 的 shortlist。
- **中期增强：** 建立控制器族图谱、拓扑覆盖地图、参考设计变换模板和基于经验的候选排序器。
- **长期技术路线：** 发展为 template synthesis + 多候选拓扑生成 + 反事实评估，让系统不仅检索参考设计，还能组合和演化参考路径。
- **解耦价值：** 用户可以只拿方案比较结果，自己继续手动设计。

### 2.3 BOM / 元器件选择 🟢

- **当前可工程化方式：** 参数化检索 + datasheet 解析 + alternates 排序，结合库存、价格、生命周期、热性能和降额约束做综合筛选。
- **Phase 0 能做什么：** 让 BOM intelligence 成为首个明确用户价值点，输出主选型、备选料、availability 风险和成本权衡。
- **中期增强：** 引入供应商偏好规则、组织级 approved vendor list、批量替代料包和供应链波动下的动态重排。
- **长期技术路线：** 演进为成本-性能-风险联合优化器，并用采购、打样、返修和测试结果持续校准排序逻辑。
- **解耦价值：** 用户可以只调用 BOM intelligence，不必生成 schematic / layout。

### 2.4 损耗与热估算 🟢

- **当前可工程化方式：** 数值计算库 + 器件参数数据库 + 模块化拓扑模型，覆盖导通损耗、开关损耗、磁芯损耗和热阻估算。
- **Phase 0 能做什么：** 覆盖常见电源家族的大部分一阶分析，给出热点器件、损耗分布、热风险和基本散热建议。
- **中期增强：** 引入 layout-aware 热估算、工况扫描、器件参数不确定性建模和仿真辅助校准。
- **长期技术路线：** 用 surrogate models 替代部分高成本热仿真，并形成跨样机、跨环境条件的多场景热优化能力。
- **解耦价值：** 用户可拿已有 BOM 或原理图直接做分析，不必从头走设计流程。

### 2.5 环路设计与稳定性分析 🟡

- **当前可工程化方式：** 模板化传递函数 + 数值求解 + 补偿网络建议，输出风险提示而非虚假的“自动最优稳定解”。
- **Phase 0 能做什么：** 对常见拓扑给出建议补偿网络、带宽范围、相位裕度风险和需要人工确认的关键假设。
- **中期增强：** 引入 simulation co-pilot、控制器族特定知识、敏感度分析和候选补偿方案的 side-by-side 对比。
- **长期技术路线：** 发展为结合测量反馈、代理模型和控制理论模板的 adaptive loop synthesis 系统。
- **解耦价值：** 用户可在分析层开始，不必先重走前面的设计流程。

### 2.6 安规、降额与规则生成 🟢

- **当前可工程化方式：** 规则表 + 条件映射 + 签核清单生成，把电压等级、隔离边界、应用场景和标准要求映射为可执行规则。
- **Phase 0 能做什么：** 生成编辑器内规则、验证约束、降额检查和 sign-off checklist，这是与通用 PCB AI 拉开差距的核心能力之一。
- **中期增强：** 支持面向行业、客户和产品线的定制规则库，并能把组织级 sign-off 习惯产品化。
- **长期技术路线：** 构建 machine-readable compliance graph，把标准、证据链、设计对象和验证结果系统化连接起来。
- **解耦价值：** 用户可对已有设计单独跑规则和 sign-off 检查。

### 2.7 原理图与 layout draft 生成 🟡

- **当前可工程化方式：** 自研编辑器中的 reference design 模板库 + 参数填充 + 支路选择 + layout 草案生成。
- **Phase 0 能做什么：** 生成 engineer-reviewable 的 schematic 与 layout draft，作为后续审查、修改和验证基础。
- **中期增强：** 增加 template synthesis、可复用 layout pattern、局部重生成和意图保持式编辑能力。
- **长期技术路线：** 发展为 conversation-driven design evolution 系统，在保留人工修改意图的前提下做增量式重构和多候选草案生成。
- **解耦价值：** 用户可以从已有 schematic / layout 开始，也可以自己手动修改后再继续。

### 2.8 Router 接入与后处理 🟢

- **当前可工程化方式：** 约束驱动的布线编排 + 电力电子专用后处理，把 placement、回流路径、间距和关键支路约束结合起来。
- **Phase 0 能做什么：** 先打通设计意图、规则约束和人工修改闭环，不承诺全自动 router 替代。
- **中期增强：** 建立面向不同拓扑的布线策略库、re-route 建议、关键网络优先级体系和人工修改感知的后处理。
- **长期技术路线：** 发展为混合人工 / AI 协同布线系统，在制造反馈和性能反馈下做多目标优化。
- **解耦价值：** 用户可以只调用布线或后处理，不必全流程自动化。

### 2.9 EMC / 电源完整性 / 高频寄生效应 🔴

- **当前可工程化方式：** 早期依赖规则引擎、经验库和风险识别，而不是承诺高置信度自动化闭环。
- **Phase 0 能做什么：** 输出风险提示、敏感环路说明、布局注意事项和建议的人工验证路径。
- **中期增强：** 引入开源求解器、SPICE / 场求解协同、寄生模型近似和高风险节点优先级分析。
- **长期技术路线：** 通过 surrogate models、lab-feedback learning 和多工况测试数据，替代部分昂贵仿真并建立高频行为预测能力。
- **解耦价值：** 用户可把系统当作 EMC / PI 风险分析助手，而非被迫依赖完整自动生成。

### 2.10 多物理场系统级闭环 🔴

- **当前可工程化方式：** 当前阶段更现实的做法是先统一数据模型、验证对象和接口边界，为后续耦合预留架构位置。
- **Phase 0 能做什么：** 不承诺自动闭环，但会保留热、EM、PI、制造和测试数据进入项目上下文的标准入口。
- **中期增强：** 把热、EMI、PI、器件漂移、制造偏差与测试结果逐步接入统一项目图谱。
- **长期技术路线：** 构建 multi-physics closed loop，让设计、仿真、测试和返修反馈形成真正的联合优化回路。
- **解耦价值：** 即使没有闭环优化，系统也可以先承担数据组织、解释和签核协作层。

### 2.11 启发式设计与设计空间探索 🟡

- **当前可工程化方式：** 多方案生成 + trade-off ranking + what-if exploration + conversation-driven revision，优先做“可讨论的候选空间”，而不是伪装成“自动最优设计器”。
- **Phase 0 能做什么：** 给出 2-3 条合理设计路径，解释各自的效率、成本、风险、器件约束、验证难度和 layout 难度，并支持对话式改目标后重新排序。
- **中期增强：** 引入反事实分析、设计图谱记忆、团队偏好学习和“为什么不选 B 方案”的结构化记录。
- **长期技术路线：** 演进为 design-space search + simulation co-pilot + active learning 系统，让 AI 在真实项目反馈中持续提升候选生成和排序质量。
- **解耦价值：** 用户可以拿现有 BOM / schematic / layout 来做替代路径探索，而不是从空白设计开始。

**核心结论：**  
Phase 0 的技术承诺依然收敛在 verified draft 与经验系统，但从中期开始，产品会逐步从“生成第一版”升级为“解释、比较、学习、优化”的工程决策平台。启发式设计正是这条长期路线中最重要的产品层和数据层抓手之一。

## 4. 数据策略（护城河）

本项目真正有价值的数据，不只是“画过多少块板”，而是**工程决策数据**。

**关键数据类型：**

- 结构化规格与约束
- 候选 reference path 与拓扑比较记录
- BOM 主选型 / 备选型及其原因
- 规则触发记录与 sign-off 证据链
- schematic / layout 的人工修改轨迹
- 启发式设计对话记录：为什么选 A、不选 B、改动背后的原因
- 仿真、实验室测试、返修与失效反馈
- 供应链波动、替代料切换和量产问题反馈

**为什么这些数据更有价值：**

- 它们比纯 CAD 文件更接近“资深工程师的判断过程”
- 它们能持续提升模型排序、规则优先级和项目助手表现
- 它们能把经验从个人知识，转化为组织级和平台级资产
- 启发式设计对话会产生“被考虑过但未被采用的方案”数据，这类反事实信息比最终结果更能帮助模型成长

**数据来源：**

- 公开 reference designs 与 datasheet
- 经授权、匿名化、聚合化的客户项目数据
- 设计服务项目中产生的高质量交互、验证与测试数据
- benchmark harness 与评测集

**治理原则：**

- 明确授权边界
- 强调匿名化与聚合化使用
- 区分公开数据、客户私有数据和平台训练数据
- 企业版与私有部署场景保留严格数据隔离能力

**飞轮路径：**

1. 先用公开资料、规则库和 reference family 建立起步能力  
2. 用 design partner / 设计服务项目获取高质量真实反馈  
3. 把人工修改、方案取舍、验证结果结构化沉淀  
4. 用这些数据持续优化排序、模板、规则和项目助手  
5. 让平台越用越懂工程设计，形成 experience system 的真实壁垒

## 5. 竞品分析

我们不假设“市场上没有 AI EDA 公司”。相反，AI EDA 已经被市场证明值得做，这对本项目是正面信号。

**现有竞品大多聚焦于：**

- 通用 PCB 协作与云端体验
- 自动布局布线
- 从自然语言到原理图或制造资料的宽流程体验
- CAD 外挂式 AI 助手

**本项目的重点则在于：**

- 电力电子中的设计决策层，而不是仅仅图形编辑层
- power-specific rules、验证和 sign-off，而不是仅仅画图自动化
- reference family、经验沉淀和项目级上下文，而不是单次生成
- 启发式设计互动，让 AI 参与方案比较、取舍解释和工程师学习
- 自研 editor + experience system，而不是停留在第三方 CAD 附加层

**这对本项目的含义：**

- “AI + PCB”已经是被验证过的赛道
- “竞争对手尚未关注 power”并非准确表述
- 我们真正的重点位于 **AI-native 编辑器、经验系统、电力电子验证层、启发式设计交互层和可解耦工作流层**

## 6. 我们的优势

我们的优势并不来自一句“我们会赢”的判断，而来自几类更难复制的系统性积累。

### 6.1 Self-built editor

- 自研 editor 让 AI 生成、人工修改、验证和版本记录处于同一工作台
- 它让 manual override 成为产品内能力，而不是工作流外行为
- 长期上，editor 是 design OS 的入口，而不仅是图形界面

### 6.2 Experience system

- 经验系统把规则库、reference family、设计逻辑、修正策略和项目上下文组织起来
- 用户购买的不只是一次生成能力，而是一套持续在线的工程经验系统
- 这使平台更接近“产品化的资深工程团队能力”

### 6.3 Power-specific verification

- 电力电子中的安规、降额、环路、热、EMC 风险和关键回流路径是高价值壁垒
- verified draft + verification pack 比“生成了一张图”更接近真实工程价值
- 高错误成本场景更容易形成深度信任和工作流嵌入

### 6.4 Heuristic design interaction

- 启发式设计让 AI 不只是自动生成器，也是方案比较器、trade-off 解释器和学习助手
- 这会显著提升用户持续使用频率，因为用户不只在“第一次生成”时需要它
- 多方案比较和反事实分析会产生比最终设计文件更宝贵的工程决策数据

### 6.5 Data flywheel

- 高价值数据来自规格澄清、方案取舍、人工修改、实验室测试与返修反馈
- 数据飞轮不仅提升模型，也提升规则、模板和项目助手
- 越多真实项目进入平台，experience system 越强

### 6.6 Service feedback loop

- 设计服务业务不是临时补收入，而是早期高质量数据和真实交付反馈的重要来源
- 它既帮助平台获得 revenue 和 design partner 背书，也帮助平台更快进入真实工程场景
- 软件与服务相互增强，形成更扎实的 wedge

### Trust Stack

Trust Stack 需要作为产品架构的正式组成部分呈现，用于说明系统的可信边界与治理方式。

**Trust Stack 包括以下要素：**

- 每一步都有 deterministic artifact  
  规格文档、候选拓扑、BOM 排序、原理图草案、layout draft、验证包均需留痕

- 每个关键决策都能追溯  
  能追溯到公式、datasheet、reference design 和规则来源，而非仅依赖模型判断

- 经验来源可以结构化沉淀  
  规则、reference、修正策略、测试结论、方案放弃理由都应沉淀为长期可复用资产

- Every stage is decouplable  
  每一步都可以独立开始 / 停止 / 重跑，不要求强制执行全自动流水线

- Manual override is first-class  
  人工修改是一级能力，并被视为标准工作流的一部分

- Heuristic exploration is governed  
  多方案探索必须附带解释、边界条件和风险提示，而不是制造“看似聪明、实则不负责任”的答案

- LLM 项目助手理解上下文  
  助手能够理解项目目标、约束和人工修改意图，并据此继续 fine-tune 建议

- human sign-off 是明确边界  
  平台生成 draft，工程师签核后才进入制造

- safety-critical 场景初期不覆盖  
  汽车 / 医疗 / 航空等高安全等级应用，不作为 Seed 阶段承诺

**制度化边界：**

- 系统允许并吸收用户手动修改
- 手动编辑被视为系统工作流的一部分，而非退出系统
- 长期大叙事不改变短期承诺边界，Phase 0 仍然聚焦 verified draft wedge

**优势的组合方式：**

- layout automation 属于组成部分，但不足以单独构成优势
- prompt UX 可以提升体验，但不足以单独构成壁垒
- 更核心的系统性优势来自  
  `self-built editor + experience system + power-specific verification + heuristic design interaction + data flywheel + service feedback loop`

## 7. 商业模式

我们采用**软件产品 + 专精电子设计服务**的双轮驱动模式，早期成功不建立在大规模 freemium 转化上。

**早期收入主线：**

- 付费 design partners 与定制设计项目
- 小团队专业订阅
- enterprise pilots 与私有部署

| 业务线 | 目标用户 | 收入形态 | 核心能力边界 |
|------|----------|----------|--------------|
| **免费版** | 学生、评估用户、社区开发者 | $0 | spec explorer、公开模板、有限 AI 额度 |
| **专业版** | 独立工程师、小团队 | $99 / 月 / 席位 | 私有项目、BOM intelligence、verified draft、导出 |
| **团队版** | 中小企业和设计公司 | $499 / 月 / 席位 | 协作、版本控制、云同步、私有规则库、更多自动化额度 |
| **企业版** | 电源公司、汽车、工业客户 | $50k-$500k / 年 | 私有部署、权限隔离、流程集成、专属支持 |
| **设计服务** | 需要快速迭代或定制开发的客户 | 项目费 / retainer | 平台辅助设计、快速迭代、定制化交付、测试与验证支持 |

**收紧免费版边界的原因：**

- 免费版可以帮助获客，但不应该承载太多私有设计价值
- 私有项目、导出、供应链集成、自动化额度和制造接口应该收费
- 这个赛道的主要收入更可能来自有明确工程价值和隐私诉求的团队
- 设计服务会成为早期验证平台价值和建立口碑的重要收入来源

### 长期收入主线

商业模式不应只被理解为几个收费档位，而应被理解为一条伴随平台能力扩展而展开的收入曲线。

**短期（Phase 0-1）：**

- design partners
- 设计服务
- 专业版与团队版订阅
- 以 verified draft、BOM intelligence、规则与验证能力变现

**中期（Phase 2-3）：**

- enterprise pilots
- private deployment
- cloud collaboration
- 专有规则库、验证模块、行业模板库
- 面向高价值行业的 workflow integration 与长期合同

**长期（Phase 4-5）：**

- engineering cloud platform
- design knowledge systems
- vertical workflow infra
- design-to-delivery services
- 供应链 / 验证 / 测试 / 制造协同收入

长期看，收入结构会从单一的 seat fee，演化为：

- 订阅收入
- 模块收入
- 工作流与集成收入
- 企业平台收入
- 服务与交付收入
- 网络化协同收入

**商业判断：**  
如果同一平台既能支持软件订阅，又能显著提升设计服务交付效率，并进一步承载企业协作、验证和交付协同，其企业价值预计将明显高于单纯按 seat 收费的通用 EDA 订阅。

`engineering platform -> design OS -> design-to-delivery stack`

## 9. 团队需求（短期 / 中期 / 长期人员规划）

我们的组织设计原则，并不是尽量压缩 headcount 本身，而是通过 AI-native workflow 提升每一个岗位的杠杆率。

**组织原则：**

- 团队规模虽会增长，但仍以 AI 放大效率为核心
- 每个岗位默认不是单点执行者，而是“AI 增强型 owner”
- 组织设计目标是减少低杠杆重复劳动，把更多精力放在高价值判断、产品化和客户交付上
- 早期团队保持克制，不代表能力不足，而代表 workflow、工具和组织密度更高

### 短期：Seed 阶段（10-15 人）

| 角色 | 人数 | 说明 |
|------|------|------|
| 电力电子 principal | 1 | 定义 reference family、规则和验证标准 |
| 编辑器 / EDA 系统工程师 | 2 | 自研 schematic / layout editor、设计生成、后处理、工具链 |
| 模型 / 工具工程师 | 1-2 | 规格澄清、排序、编排、benchmark harness、项目助手 |
| 云端 / 产品工程师 | 1 | 协作、同步、版本控制、工作流与用户体验 |
| 设计服务 / 应用工程师 | 1-2 | 快速交付、客户定制、测试反馈回流 |
| founder-led GTM / design partner | 1 | 客户验证、试点推进、融资叙事 |

**组织目标：**

- 做出 editor + AI + service loop 的第一条可信工作流
- 用 AI-first workflow 把 Phase 0 wedge 打穿
- 即使只有 10-15 人，也能够同时覆盖 editor、AI、设计服务、验证、内容沉淀和 GTM

### 中期：Phase 1-3（80-100 人）

**关键新增岗位：**

- 更多 domain principals（拓扑、磁件、控制、EMC、热）
- 仿真 / 数值分析工程师
- 平台 / 云基础设施工程师
- 企业产品与权限治理工程师
- 解决方案工程师 / 客户成功
- 设计服务交付负责人

**组织目标：**

- 从单一 wedge 扩展到更多拓扑与行业
- 支撑 cloud collaboration、enterprise pilot、private deployment
- 让产品、平台和交付三条线同时成立
- 形成平台化扩张后的多职能高杠杆组织，而不是传统大团队堆叠

### 长期：Phase 4-5（400-500 人）

**关键新增岗位：**

- multi-physics / surrogate model 研究与工程团队
- manufacturing integration / delivery orchestration 团队
- enterprise sales / solutions / operations
- 行业解决方案与生态合作团队
- 测试、验证、样机与交付协同团队

**组织目标：**

- 从 engineering product company 演进到 engineering platform company
- 支撑 design OS 和 design-to-delivery stack 的平台化运营
- 即使进入 400-500 人规模，公司仍保持 AI-native 组织特征，规模主要服务于平台、行业、企业交付和 design-to-delivery 网络，而不是低效率扩员

**团队判断：**  
Seed 阶段保持组织克制是必要的，但克制并不意味着能力边界很窄，而意味着公司会通过 AI-native workflow 把每个人的能力放大到更高杠杆。长期上，如果目标是 design OS 与 design-to-delivery stack，那么人才结构必须从“做工具”逐步扩展为“做平台、做交付、做行业基础设施”；人数的增长，是为了承接更广的平台、行业和交付网络，而不是因为组织效率不足。

## 11. Seed 阶段更像样的里程碑

相比于“1000 个成功制造 PCB”，更适合作为 Seed 阶段里程碑的是质量型指标和工作流型指标。

**建议里程碑：**

- 5 个付费 design partners
- 20 个 benchmark reference designs 跑通
- 50 个 engineer-reviewed 设计包
- 20 个成功点亮或无重大返工的板
- `spec -> BOM + schematic` 的中位时间
- layout draft 的人工二次修改时长
- 中级工程师使用平台后接近资深工程师 baseline 的 first-pass quality
- 启发式设计产生的候选方案中，被工程师真正采纳的比例
- 用户在方案讨论后修改目标、系统建议能否一致更新
- 任意阶段启动能力是否稳定
- 人工修改后继续编排是否顺畅

**采用这些指标的原因：**

- 它们更早反映真实工程价值
- 它们更能支撑下一轮融资叙事
- 它们比单纯“制造数量”更能体现 first-pass quality
- 它们能验证启发式设计是否真的帮助工程师拓宽设计思路，而不是只制造噪音

## 12. 当前建议默认策略选择

这部分不再写成模糊未决项，而给出当前默认答案。

1. **产品形态：自研 AI-native 编辑器 / all-in-one app first**  
   从 Day 1 建设自有编辑器，首版覆盖最有价值的关键步骤，并保留手动接管能力。

2. **MVP 切入点：Buck DC-DC first**  
   最可重复、最可验证、最容易形成闭环。

3. **首个交付物：verified draft first**  
   先交付可审查、可验证、可继续编辑的第一版设计草案，而不是一键生产级结果。

4. **设计理念：启发式设计 first-class**  
   AI 不只是自动生成器，而是方案比较器、trade-off 解释器和工程师学习助手。

5. **公司形态：软件产品 + 设计服务双轮驱动**  
   软件提供规模化交付，设计服务提供高价值收入、真实项目反馈和经验沉淀。

6. **行业重点：power electronics first，随后进入高价值行业**  
   重点覆盖电车、数据中心、芯片、储能、通信等复杂度高、付费能力强的领域。

7. **地理重点：GTM 北美，工程 / 制造联动放华南**  
   早期愿意为效率和风险控制付费的客户更可能来自北美；工程和制造联动仍可深度连接中国。

8. **长期平台方向：engineering platform -> design OS -> design-to-delivery stack**  
   先完成电子设计平台，再扩展到机电协同、团队云协作和更完整的交付基础设施。

## 13. 风险与应对

| 风险 | 严重度 | 应对方式 |
|------|--------|----------|
| 自研编辑器范围过大 | 🔴 高 | 首版只覆盖关键步骤，把完整 end-to-end 留给后续阶段逐步扩展 |
| 模型产出不稳定，工程师不信任 | 🔴 高 | 每一步附 deterministic artifact 和 verification pack，保持 human sign-off |
| 经验系统难以结构化 | 🔴 高 | 通过规则库、reference family、设计服务项目和修正轨迹逐步沉淀 |
| 高质量数据不足 | 🔴 高 | 依赖 reference、benchmark harness、设计服务项目、人工修正轨迹和 opt-in 数据 |
| 启发式设计带来信息过载 | 🟡 中 | 默认输出有限候选数、明确 trade-off 和边界条件，避免“多方案但无判断” |
| 用户觉得流程太复杂 | 🟡 中 | 把 decoupled workflow 做成一级能力，允许用户只用其中一段 |
| LLM 助手不能正确吸收人工修改 | 🟡 中 | 把项目上下文和修改历史作为明确系统层，减少对隐式记忆的依赖 |
| 长期愿景被误解为短期承诺 | 🟡 中 | 在文档中区分 Phase 承诺与远期方向，避免用大叙事替代近期交付 |
| 企业版和私有部署复杂度超预期 | 🟡 中 | 先从高价值 pilot 做起，用有限定制验证权限、治理和隔离需求 |

## 14. 面向投资人的核心表述（一分钟版本）

> 工程设计长期依赖经验，这也是资深工程师和大公司体系持续占优的原因。更稀缺的能力，在于把这些分散的工程经验组织成可持续运行的软件系统。  
>
> 本项目定位为面向电力电子的 AI-native 自研设计编辑器 / 工程平台。系统从规格出发，优先基于成熟 reference design 生成可审查、可验证、可制造的第一版设计草案，并附带 verification pack 供工程师签核。  
>
> 用户购买的并不只是单一软件工具，而是一套持续在线的 experience system。对小公司和小团队而言，这相当于获得接近顶级工程组织的规则体系、方法论和设计决策支持。  
>
> 我们的 AI 不只负责自动生成，也承担启发式设计角色：它会主动提出多个设计路径、解释 trade-off、帮助工程师拓宽思路，并把“为什么选这个方案”沉淀为项目记忆的一部分。  
>
> 该系统不要求用户执行强制全自动流水线。用户可以在自研编辑器内从任意阶段进入，在任意阶段暂停，并对 schematic、PCB layout、BOM 或规则进行手动修改，之后再交回系统继续推进。  
>
> LLM 项目助手贯穿项目周期的协作层：它维护约束、解释取舍、吸收人工修改，并支持工程师持续 fine-tune 设计结果。  
>
> 长期壁垒主要来自 experience system、经授权的数据飞轮、启发式设计交互层，以及从规格、reference、规则、验证、人工 fine-tune 到实测反馈的持续改进回路。  
>
> 公司的阶段性路径，是先建立可信的 verified draft wedge，再升级为 design OS，长期延伸到 design-to-delivery stack。

## 15. 验证计划

仅依靠展示性质的 demo 并不足以支撑论证。更合理的验证方式，是建立一套能够与人工流程和参考设计对比的 benchmark。

**验证主轴：**

1. 早期以 **自研编辑器中的 reference-design adaptation** 作为首个承诺路径
2. 首个交付重点是 **verified draft + experience system**
3. 阶段性重点放在 **first-pass quality、经验复用能力、启发式设计有效性与数据回流能力**

**新增的验证目标：**

- 任意阶段启动能力是否可用
- 人工修改后继续编排是否稳定
- LLM 助手是否能正确吸收项目上下文与修改意图
- 启发式设计是否真的帮助工程师看见更优候选空间
- 多方案讨论后，系统是否能一致更新建议和验证逻辑

**具体 benchmark 维度：**

- reference design benchmark 是否跑通
- first-pass quality 是否足够高
- 人工修改量是否显著下降
- BOM intelligence 是否明显优于人工初筛
- 点亮 / 返工 / 验证包完整度是否达到可用水平
- 中途接管后恢复执行是否稳定
- conversation-driven fine-tuning 后结果是否一致更新
- 中级工程师是否能在平台辅助下接近资深工程师 baseline
- 设计服务项目产生的数据是否持续改善模型、规则和项目助手表现
- 启发式设计产生的替代路径中，有多少被工程师认为“本来不会优先想到但值得认真评估”

**建议的验证路径：**

1. **最小可行产品演示**  
   选同步 Buck reference family，给定规格，在自研编辑器中自动生成 verified draft。

2. **Reference benchmark 套件**  
   选 20 个真实参考设计，跑完整流程，对比：
   - `spec -> BOM + schematic` 时间
   - layout draft 的人工修改时长
   - BOM 成本与替代料质量
   - 点亮率与重大返工数量
   - verification pack 完整度

3. **解耦工作流测试场景**  
   - 从已有 reference design 开始，只做 BOM 和验证
   - 从已有 schematic 开始，只做 layout draft 和验证包
   - 用户手动改 layout 后，再让系统补充验证与风险提示
   - 用户暂停在 BOM 阶段，数天后恢复并继续设计
   - 用户与助手对话修改目标后，系统建议是否同步更新

4. **启发式设计测试场景**  
   - 给同一规格生成多个候选路径，观察工程师是否认可 trade-off 解释
   - 修改目标优先级后，验证候选排序是否合理变化
   - 记录“原本不会想到的方案”是否真正进入工程师评估清单

5. **记录人工修正和失败案例**  
   把“为什么改”“为什么失败”“为什么放弃某个候选方案”结构化，这比单纯收集板子文件更值钱。

6. **争取真实制造与测试反馈**  
   一旦进入打样、点亮、返修、热像、EMI 测试，就开始建立真正的闭环壁垒。

**更关键的验证问题包括：**

- 中位速度是否有数量级提升
- 第一版设计是否足够接近可交付水平
- 修改量是否显著下降
- 用户是否真的愿意在系统里反复协作和 fine-tune
- 启发式设计是否真的提升了方案质量，而不是仅增加讨论成本
- 系统是否在真实工程反馈中持续变好
- 平台是否真的缩小了小团队与大公司之间的经验差距

**结尾判断：**  
短期我们验证的是 Buck / verified draft / reference-design adaptation first 的 wedge；中期我们建立的是经验系统和 design OS；长期我们追求的是把平台延伸到 design-to-delivery stack。文档的长期蓝图可以很大，但近期承诺依然清晰：先用可信工作流建立信任，再把这套工作流扩展成真正的工程基础设施。
